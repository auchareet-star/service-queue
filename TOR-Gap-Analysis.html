Gap Analysis — TOR Smart Queue (โรงพยาบาล)

เทียบความต้องการใน "Draft TOR — Smart Queue" กับความสามารถของระบบ HAOS Queue ปัจจุบัน

วันที่วิเคราะห์: 2026-06-26 · อ้างอิง source code + เอกสาร Flow ระบบ

บทสรุปผู้บริหาร

TOR ฉบับนี้เป็นสเปก ระบบบริหารคิวโรงพยาบาล (Hospital Smart Queue) ที่ผูกแน่นกับระบบสารสนเทศโรงพยาบาล (HIS), การตรวจสอบสิทธิ์ สปสช., บัตรประชาชน, การนัดหมาย, แจ้งเตือนผ่าน Line OA, ชำระเงินออนไลน์, จ่ายยา/ส่งยา และ Tele-pharmacy พร้อม ฮาร์ดแวร์ตู้ KIOSK/Check-in

ระบบ HAOS Queue ปัจจุบันเป็น "แกนกลางจัดการคิวทั่วไป" (General Queue Engine) ที่แข็งแรงในเรื่อง: การออกบัตรคิว, เช็คอิน, เรียก/ส่งต่อ/พัก/เรียกซ้ำ, จอแสดงผล TV, และอัปเดต Real-time ผ่าน SignalR

สรุปสั้น: ระบบปัจจุบันครอบคลุม "กลไกคิว" (calling / display / check-in / real-time) ได้ดี — ตรงกับข้อ 3.3, 3.4, 3.6 บางส่วน และข้อ WebSocket ได้เต็ม ✅  |  แต่ ยังขาดทั้งหมด ในส่วนที่เป็น "หัวใจทางการแพทย์/โรงพยาบาล": HIS, สปสช., บัตรประชาชน, ข้อมูลผู้ป่วย/Visit, นัดหมาย (3.5), Line OA (3.7), ชำระเงิน/จ่ายยา (3.8), Tele-pharmacy (3.9)
ประเด็นสำคัญ: ฟีเจอร์ที่ "ยังไม่มี" ส่วนใหญ่ ไม่ใช่งานปรับเล็กน้อย แต่เป็นโมดูลใหม่ที่ต้องเชื่อมต่อระบบภายนอก (HIS API Gateway, NHSO, ID Card Reader, LINE OA, Payment Gateway) ซึ่งต้องอาศัยความร่วมมือ/สเปก API จากโรงพยาบาลและ Third party

3.1 โปรแกรมคัดกรอง ตรวจสอบสิทธิ์ และข้อมูลผู้ป่วย

ข้อความต้องการตาม TORสถานะระบบทำได้แบบไหน / ต้องพัฒนาอะไร
3.1.1คัดแยกสถานะผู้ป่วยตามสิทธิ์และความเร่งด่วนยังไม่ได้มี ServiceGroup/ประเภทบริการ แต่ไม่มีระบบสิทธิ์ (entitlement) และคิวเร่งด่วน (priority/triage) — ต้องเพิ่มฟิลด์ความเร่งด่วน + ตรรกะคิวลัด
3.1.2ตรวจสอบสิทธิ์ สปสช. ผ่านบัตรประชาชน + บันทึกรูปถ่ายยังไม่ได้ไม่มีการเชื่อม NHSO, ไม่มี ID Card Reader integration, ไม่มีถ่ายรูป — ต้องพัฒนาใหม่ทั้งหมด
3.1.3ส่งข้อมูลเปิดลงทะเบียนผ่าน HIS (API Gateway)ยังไม่ได้ไม่มี integration กับ HIS — ต้องสร้าง connector ตามสเปก API ของโรงพยาบาล
3.1.4ตรวจประวัติผู้ป่วย + เลือกคลินิก + เปิด Visitยังไม่ได้ไม่มีแนวคิด Patient/Visit ในระบบ (มีเฉพาะ Queue/Service) — ต้องเพิ่มโมเดลผู้ป่วยและ Visit
3.1.5แสดงสิทธิ์ที่ใช้ได้ (บัตรทอง/ข้าราชการ/ชำระเอง) ให้เลือกยังไม่ได้ไม่มีข้อมูลสิทธิ์ — ขึ้นกับการเชื่อม สปสช./HIS
3.1.6พิมพ์บัตรคิว (รหัสการรักษา/ข้อมูลผู้ป่วย/QR) ปรับแต่งโดยผู้ดูแลบางส่วน✔ ระบบออกบัตรคิว + QueueKey (GUID) สำหรับ QR ได้  ✘ แต่บัตรมีแค่เลขคิว ไม่มีข้อมูลผู้ป่วย และ layout ปรับแต่งเองยังจำกัด — ต่อยอดจาก report engine (Stimulsoft) ได้
3.1.7-8ตรวจประวัติผู้ป่วย + เลือกคลินิก + เปิด Visitยังไม่ได้เหมือน 3.1.4 — ต้องมีโมเดลผู้ป่วย/Visit + เชื่อม HIS
3.1.9เปิด Visit จากนัดหมายหลายคลินิกในวันเดียวกันยังไม่ได้ไม่มีโมดูลนัดหมาย (ดู 3.5) และ Visit
3.1.10ตรวจนัด Lab/X-ray/หัตถการ ที่ผูกกับนัดก่อนหน้าอัตโนมัติยังไม่ได้ต้องเชื่อม HIS/LIS/RIS
3.1.11รองรับ WebSocket ลดภาระดึงข้อมูล/จัดลำดับคิวซ้ำทำได้✔ ตรงสเปก — ระบบใช้ SignalR (WebSocket) ผ่าน EventHub ยิง event แบบ push (TriggerQueueChange/Calling) อยู่แล้ว

3.2 โปรแกรมลงทะเบียนผู้ป่วยใหม่

ข้อความต้องการตาม TORสถานะหมายเหตุ
3.2.1ตรวจว่าผู้ป่วยมีประวัติการรักษาแล้วหรือยังยังไม่ได้ไม่มีฐานข้อมูลผู้ป่วย — ต้องเชื่อม HIS
3.2.2-3บันทึกข้อมูลลงทะเบียน + ยืนยันก่อนบันทึกยังไม่ได้ไม่มีฟอร์มลงทะเบียนผู้ป่วย — ต้องพัฒนาโมดูลใหม่
3.2.4ส่งข้อมูลลงทะเบียนผ่าน HIS (API Gateway)ยังไม่ได้ต้องสร้าง HIS connector
3.2.5ส่งข้อมูลยืนยันการลงทะเบียนผ่าน Line OAยังไม่ได้ไม่มี Line OA integration
3.2.6แสดงข้อมูลให้เจ้าหน้าที่ที่เกี่ยวข้องรับทราบบางส่วน✔ มีหน้าจอเจ้าหน้าที่ + แจ้งแบบ real-time อยู่แล้ว — แต่ต้องผูกกับข้อมูลผู้ป่วยที่ยังไม่มี
3.2.7รองรับ Responsive ทั้ง Mobile และ Monitorทำได้✔ Frontend เป็น Angular + responsive และมีหน้า Mobile (/mobile/*) อยู่แล้ว

3.3 โปรแกรมเรียกคิวผู้ป่วย (โมดูลที่ระบบใกล้เคียงที่สุด)

ข้อความต้องการตาม TORสถานะระบบทำได้แบบไหน
3.3.1เรียกคิวตามลำดับเวลามา / เวลาถึงจุดบริการ / เวลานัดบางส่วน✔ เรียกแบบ FIFO ตามเวลาเช็คอินได้ (NextAsync auto-select เก่าสุด)  ✘ "ตามเวลานัด" ยังไม่มี (ไม่มีโมดูลนัด)
3.3.2เรียกลัดคิวตามความเร่งด่วนอัตโนมัติยังไม่ได้ไม่มี priority/urgency — ต้องเพิ่มลำดับความสำคัญและตรรกะคิวลัด
3.3.3ค้นหาผู้ป่วยเพื่อเรียกคิวแบบเฉพาะเจาะจงบางส่วน✔ เรียกคิวเฉพาะเจาะจงได้ด้วย QueueId/เลขคิว  ✘ แต่ค้นจาก "ข้อมูลผู้ป่วย" (ชื่อ/HN) ยังไม่ได้
3.3.4เรียกคิวตามตรรกะที่โรงพยาบาลกำหนดบางส่วน✔ รองรับ FIFO + เรียกเอง + ส่งต่อ (Forward) ข้ามห้อง/บริการ  ✘ ตรรกะซับซ้อนเฉพาะทางต้องปรับเพิ่ม
3.3.5แสดงรายชื่อผู้ป่วยที่รอ ตามช่วงเวลา/ความเร่งด่วน/สถานะบางส่วน✔ แสดงรายการคิวแยกตามสถานะ (CheckIn/Holding/Inprogress) + เวลารอ  ✘ จัดตามความเร่งด่วนยังไม่ได้
3.3.6พิมพ์ใบนำทาง (Navigation slip) ตามรูปแบบที่กำหนดบางส่วน✔ มี report engine (Stimulsoft, ExportReportPdf) ที่ต่อยอดพิมพ์ใบนำทางได้  ✘ ยังไม่มี template ใบนำทางเฉพาะ
3.3.7แสดงสถานะ LAB/X-Ray + ล็อค/ปลดล็อคคิวอัตโนมัติยังไม่ได้ต้องเชื่อม LIS/RIS และเพิ่มกลไกล็อคคิวตามสถานะผล
3.3.8แถบสีสถานะตามแผนก + สถานะ (รอ/กำลังรักษา/เรียกไปแล้ว)บางส่วน✔ มีสถานะคิว (None/CheckIn/Inprogress/Holding/Completed) + แสดงคิวเรียกข้าม (Skipped) บนจอ TV  ✘ แถบสีแยกตามแผนกต้องปรับ UI
3.3.9ฟังก์ชันเรียกคิวห้องยา (ประวัติยา/แพ้ยา/Vital/Note/Label/แจ้งแพทย์)ยังไม่ได้เป็นฟีเจอร์เภสัชกรรมเชิงลึก ต้องเชื่อม HIS/Pharmacy — ไม่มีในระบบ
3.3.10ห้องฉุกเฉิน ปรับความเร่งด่วนตาม Emergency Triage Scoreยังไม่ได้ต้องมีระบบ triage/priority

3.4 โปรแกรมจอแสดงคิวผู้ป่วย (จุดแข็งของระบบ)

ข้อความต้องการตาม TORสถานะระบบทำได้แบบไหน
3.4.1ปรับรูปแบบหน้าจอเหมาะแต่ละแผนก โดยผู้ดูแลบางส่วน✔ TV ตั้งค่า work list (บริการ+ห้อง+เคาน์เตอร์) ที่จะแสดงได้  ✘ ปรับ layout/ธีมเชิงลึกยังจำกัด
3.4.2แสดงคิวที่กำลังรอ / กำลังรักษา / ถูกเรียกข้ามไปแล้วทำได้✔ จอ TV (ScreeningPointRoom) แสดงคิวปัจจุบันที่แต่ละเคาน์เตอร์ + รายการคิวที่ถูกข้าม (Skipped) ครบ
3.4.3แสดงเลขคิว ชื่อ-สกุล เวลารอ ความเร่งด่วน (อย่างน้อย)บางส่วน✔ แสดงเลขคิว + ชื่อ + เวลารอได้  ✘ "ความเร่งด่วน" และชื่อผู้ป่วยจริงต้องผูก patient data
3.4.4แสดงผู้ป่วยรอหลายแผนกใน 1 จอทำได้✔ TV รองรับหลายบริการ/ห้อง + carousel เคาน์เตอร์
3.4.5หน้าจอ Dashboard ภาพรวมจำนวนผู้ป่วยทำได้✔ มี Dashboard (get-dashboard) + เกณฑ์สีเวลารอ (Configuration) อยู่แล้ว
3.4.6จอคิวห้องยา/การเงิน แสดงสถานะ (กำลังจัดยา/รอการเงิน ฯลฯ)บางส่วน✔ กลไกสถานะคิว + จอแสดงผลมีอยู่  ✘ สถานะเฉพาะห้องยา/การเงินต้องเพิ่ม status ใหม่

3.5 โปรแกรมนัดหมายผู้ป่วย (เจ้าหน้าที่ + ออนไลน์)

ทั้งโมดูลยังไม่มีในระบบ — ระบบ HAOS Queue ปัจจุบันไม่มีแนวคิด "นัดหมาย (Appointment)" ทั้งฝั่งเจ้าหน้าที่และฝั่งผู้ป่วยออนไลน์ ต้องพัฒนาเป็นโมดูลใหม่ทั้งหมด
ข้อความต้องการตาม TORสถานะหมายเหตุ
3.5.1.xนัดหมายสำหรับเจ้าหน้าที่ (กำหนดเวลารับ, เชื่อมระบบนัด รพ., จองตามแพทย์/แผนก, active อัตโนมัติ, ยืนยันผ่าน Line OA)ยังไม่ได้ต้องสร้างโมดูล Appointment + เชื่อม HIS + Line OA
3.5.2.xนัดหมายออนไลน์สำหรับผู้ป่วย (ลงทะเบียน, ยืนยันตัวตน, จองแผนก/แพทย์/เวลา, ดูประวัติ QR, ตรวจสิทธิ์, ผูก Line OA/เว็บไซต์)ยังไม่ได้ต้องสร้าง portal/แอปจองออนไลน์ + ยืนยันตัวตน + เชื่อม Line OA

3.6 โปรแกรมยืนยันหน้าแผนกก่อนเข้ารับบริการ

ข้อความต้องการตาม TORสถานะระบบทำได้แบบไหน
3.6.1ลงทะเบียนเข้าคิวการรักษา (Check-in)ทำได้✔ มี CheckinAsync + เช็คอินผ่าน QR (get-queue-scancheckin) อยู่แล้ว
3.6.2ตรวจสอบสิทธิ์ สปสช. ผ่านบัตรประชาชนยังไม่ได้ไม่มี NHSO / ID card — เหมือน 3.1.2
3.6.3ส่งข้อมูลเปิดลงทะเบียนผ่าน HISยังไม่ได้ต้อง HIS connector
3.6.4-5ตรวจประวัติ + เปิด Visit + แสดงสิทธิ์ยังไม่ได้ต้องมี Patient/Visit + สิทธิ์
3.6.6พิมพ์บัตรคิว + QR + ปรับแต่งได้บางส่วน✔ ออกบัตร + QR ได้ ✘ ข้อมูลผู้ป่วย/ปรับ layout จำกัด (เหมือน 3.1.6)
3.6.7-8เปิด Visit หลายคลินิก + ตรวจนัด Lab/X-rayยังไม่ได้ต้องเชื่อม HIS/นัดหมาย
3.6.9รองรับ WebSocket ลดภาระดึงข้อมูลทำได้✔ SignalR (เหมือน 3.1.11)
3.6.10ตรวจสอบลำดับคิวการรักษาทำได้✔ GetQueueDetail คืน จำนวนคิวที่รออยู่ก่อนหน้า (WaitingQueue) + ติดตามผ่าน Mobile ได้

3.7 ระบบแจ้งเตือนการรักษา (Line OA)

ทั้งโมดูลยังไม่มี — ไม่มีการเชื่อม LINE OA / Flex Message / Auto message / ตั้งเวลาส่ง ต้องพัฒนาใหม่ (LINE Messaging API)
ข้อความต้องการสถานะ
3.7.1-5แจ้งเตือนการรักษา/ผล/นัด ผ่าน Line OA, Flex Message, ตั้งเวลาส่ง, บันทึกสนทนา, Auto messageยังไม่ได้

3.8 ระบบชำระเงินออนไลน์ & จ่ายยา (Smart Dispensing)

ทั้งโมดูลยังไม่มี — ไม่มีระบบชำระเงิน, เชื่อมเภสัชกรรม, จัดส่งยา, Pickup Code, รายงานการจัดส่ง ต้องพัฒนาใหม่ + เชื่อม Payment Gateway/Pharmacy

ข้อสรุป & ข้อเสนอแนะ

สิ่งที่ระบบทำได้ทันที (ใช้เป็นฐานต่อยอด)

  • ✅ แกนเรียกคิว: เรียก/ส่งต่อ/พัก/เรียกซ้ำ/เสร็จสิ้น (FIFO + เลือกคิวเฉพาะ)
  • ✅ เช็คอิน (manual + QR scan) + ตรวจลำดับคิว/จำนวนคิวรอ
  • ✅ จอแสดงผล TV: คิวปัจจุบัน/เรียกข้าม, หลายแผนกใน 1 จอ, เสียงเรียก
  • ✅ Dashboard + เกณฑ์สีเวลารอ
  • ✅ Real-time WebSocket (SignalR) — ตรงสเปกข้อ 3.1.11 / 3.6.9
  • ✅ Responsive + Mobile tracking ผ่าน QueueKey/QR
  • ✅ Report engine (Stimulsoft PDF) ต่อยอดบัตรคิว/ใบนำทางได้

สิ่งที่ต้องพัฒนาใหม่ (Gap หลัก เรียงตามความสำคัญ)

  1. 🔌 HIS Integration (API Gateway) — รากฐานของเกือบทุกข้อ (ผู้ป่วย, Visit, สิทธิ์, นัด)
  2. 🆔 ตรวจสอบสิทธิ์ สปสช. + เครื่องอ่านบัตรประชาชน + ถ่ายรูป
  3. 👤 โมเดลผู้ป่วย/Visit + ลงทะเบียนผู้ป่วยใหม่ + ระบบสิทธิ์การรักษา
  4. ⚡ ระบบความเร่งด่วน/Triage (คิวลัด, Emergency Triage Score)
  5. 📅 โมดูลนัดหมาย (เจ้าหน้าที่ + ออนไลน์)
  6. 💬 LINE OA (แจ้งเตือน, Flex Message, ยืนยันนัด, Pickup Code)
  7. 💊 เภสัชกรรม: เรียกคิวห้องยา (ประวัติยา/แพ้ยา/Vital), สถานะ LAB/X-ray
  8. 💳 ชำระเงินออนไลน์ + Smart Dispensing + Tele-pharmacy
  9. 🖥️ ฮาร์ดแวร์ KIOSK/Check-in + หน้าจอ kiosk + driver อุปกรณ์

Integration, (2) Patient/Visit/Appointment, (3) LINE OA + Notification, (4) Payment + Pharmacy/Tele-pharmacy — พร้อมจัดหาฮาร์ดแวร์ตู้บริการ
