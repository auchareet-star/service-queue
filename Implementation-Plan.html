1. ภาพรวม & หลักการวางแผน

แผนนี้ยึด 3 หลักการเพื่อลดความเสี่ยงและส่งมอบคุณค่าเร็ว:

  1. MVP-first / ส่งมอบเป็นเฟส — เปิดใช้ส่วนที่ทำได้ก่อน ไม่รอครบทุกอย่าง
  2. แยกงานที่ไม่พึ่ง HIS ออกมาทำก่อน — เฟส 1 เดินได้ทันทีโดยไม่ต้องรอสเปก HIS (ลด blocking)
  3. งานที่พึ่งระบบภายนอก (HIS/สปสช./LINE/Payment) = critical path — เริ่มเคลียร์สเปกตั้งแต่เฟส 0
จุดแข็งที่มีอยู่แล้ว (ไม่ต้องสร้างใหม่): แกนเรียกคิว, เช็คอิน, Forward, จอ TV, Dashboard, SignalR real-time, report engine — เฟส 1 ส่วนใหญ่เป็นการ ต่อยอด ไม่ใช่สร้างจากศูนย์

3. Dependency & Critical Path

เส้นทางวิกฤต (Critical Path): งานที่ต้องเริ่มก่อนและบล็อกงานอื่นมากที่สุด

Dependency ภายนอกกระทบเฟสถ้าไม่พร้อม
HIS API spec + sandbox2, 3, 4เลื่อนเฟส 2 เป็นต้นไป — เฟส 1 ยังเดินได้
NHSO (สปสช.) access2ใช้เครื่อง สปสช. แยก/เลือกสิทธิ์เอง (ทางเลือกในเอกสาร HIS)
LINE OA channel3เลื่อนแจ้งเตือน — โมดูลนัดหมายในระบบยังทำได้
Payment Gateway4ใช้แคชเชียร์เดิม + manual mark paid
ฮาร์ดแวร์ KIOSK/Check-in1, 5พัฒนา UI บนเครื่องทดสอบไปก่อน รอของมาติดตั้ง
กลยุทธ์ลดความเสี่ยง: เริ่ม เฟส 0 (เคลียร์สเปก HIS) + เฟส 1 (งานไม่พึ่ง HIS) พร้อมกันตั้งแต่วันแรก — ทำให้มีของส่งมอบเร็วและไม่เสียเวลารอ HIS

4. ความเสี่ยงหลัก & ข้อแนะนำ

ความเสี่ยงผลกระทบวิธีรับมือ
HIS ไม่เปิด API หรือเปิดช้าสูงระบุใน TOR/สัญญาว่าโรงพยาบาลต้องจัดเตรียม API + ใช้ทางเลือก (reverse-flow/manual/deep-link)
Scope เต็มไม่ทันใน 120 วันสูงแบ่งส่งมอบเป็นเฟส — MVP (เฟส 0–1) ใน 120 วัน เฟสที่เหลือเป็นงวดถัดไป
ข้อมูลสุขภาพ/PDPAกลางDeep-link แทนดึงข้อมูล, audit log, data minimization (ดูเอกสาร HIS §7)
Routing วนลูปไม่รู้จบกลางกลไก mark order completed + จำกัดรอบ (ดู Service Routing §8)
หลายระบบภายนอกในเฟส 4สูงทำเฟส 4 ท้ายสุด หลังแกนนิ่ง + เลือกเฉพาะที่อยู่ใน scope
ข้อแนะนำสำคัญ: เจรจาขอบเขตสัญญาให้ แยกชัดระหว่าง "ฮาร์ดแวร์ + ระบบคิวแกน (ส่งใน 120 วัน)" กับ "โมดูลเชื่อม HIS/นัด/ชำระเงิน (งวดถัดไป)" พร้อมระบุ dependency ที่โรงพยาบาลต้องจัดเตรียม — เพื่อกันงานบานปลาย/ส่งไม่ทัน
