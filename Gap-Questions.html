❓ คำถาม Gap — สำหรับคุยกับโรงพยาบาล / ทีม HIS

รวมคำถามที่ต้องเคลียร์ก่อนเริ่มพัฒนา — เพื่อกำหนดขอบเขต ลดความเสี่ยง และประเมินงานได้แม่นยำ

ขยายจากร่างคำถามเดิม + เพิ่มหัวข้อที่ยังขาด · 2026-06-26

ตอบคำถามคุณ: "แบบนี้พอได้มั้ย" — ร่างเดิม 10 ข้อ ดีและตรงจุดมาก โดยเฉพาะข้อขอบเขต "ระบบที่ไม่เกี่ยวกับคิว" ผมจัดกลุ่มใหม่ + เพิ่มคำถามย่อยให้คม และเติมหัวข้อที่ขาด (ปริมาณ/Performance, ฮาร์ดแวร์/จอ, PDPA, ส่งมอบ/migration)   ของคุณ = คำถามจากร่างเดิม   เพิ่มเติม = ที่ผมเสนอเพิ่ม

A · ขอบเขตงาน (Scope) — สำคัญสุด ถามก่อนเลย

กำหนดให้ชัดว่าอะไร "สร้าง / เชื่อมต่อ / นอกขอบเขต" — กันงานบานปลาย
A1
ระบบที่ไม่เกี่ยวกับคิว (HIS, การเงิน, เภสัช, LIS/RIS) เราต้อง "พัฒนา" หรือ "เชื่อมต่อระบบเดิม" หรือ "นอกขอบเขต"?ของคุณ
  • ระบบไหนเป็นของเดิมที่เราแค่ "ต่อ API"?
  • ระบบไหนต้องสร้างส่วนติดต่อ/หน้าจอใหม่?
  • ส่วนไหน "นอกขอบเขต" ชัดเจน (ไม่ทำ)?
ทำไม: ตัวกำหนด scope/ราคา/ความเสี่ยงทั้งโครงการ — ถ้าไม่ชัด งานจะบานปลาย

B · การเชื่อมต่อ HIS

B1
เชื่อมต่อรูปแบบไหน? (REST/JSON, HL7 v2, FHIR) มี API Gateway/middleware อยู่แล้วไหม?ของคุณ
  • HIS ยี่ห้อ/เวอร์ชันอะไร (HOSxP/JHCIS/SSB/Vendor เฉพาะ)? ใครดูแล? ติดต่อใคร?
  • รองรับ webhook/event push หรือทำได้แค่ให้ polling?
  • Auth แบบไหน (OAuth2/API key/mTLS)? มี IP allowlist?
  • มี sandbox/test environment ไหม? พร้อมเมื่อไร?
  • SLA/uptime, rate limit, ปริมาณ request สูงสุด?
ทำไม: กำหนดสถาปัตยกรรม (sync/async), effort การ map, และ เป็น critical path — ถ้า HIS ไม่พร้อมงาน block

C · ลงทะเบียนผู้ป่วยใหม่

C1
ลงทะเบียนผู้ป่วยใหม่ต้องใช้ข้อมูล (ฟิลด์) อะไรบ้าง?ของคุณ
  • ฟิลด์บังคับ vs ไม่บังคับ (CID, คำนำหน้า, ชื่อ-สกุล, วันเกิด, เพศ, ที่อยู่, เบอร์, สิทธิ์...)?
  • ระบบคิว "เขียน" ข้อมูลเข้า HIS ได้ไหม หรือเจ้าหน้าที่ลงใน HIS แล้วเราอ่าน HN กลับ? (รพ.ส่วนใหญ่ไม่ให้เขียนตรง)
  • ตรวจซ้ำ (duplicate) ยังไง — ใช้ CID เป็น key?
  • ต้องบันทึก รูปถ่าย/ภาพบัตร ไหม? เก็บที่ฝั่งไหน?
ทำไม: กำหนดฟอร์ม + ทิศทางข้อมูล (read/write) + ความเสี่ยง PDPA

D · Visit

D1
เปิด Visit ต้องใช้/ส่งข้อมูลอะไรบ้าง?ของคุณ
  • ฟิลด์ที่ต้องส่งตอนเปิด Visit (HN, clinic code, eligibility, doctor, visit type, วันที่...)?
  • ระบบคิว เปิด Visit เองได้ไหม หรือ HIS เปิดแล้วส่ง VN กลับ (reverse-flow)?
  • 1 ผู้ป่วยมีหลาย Visit/วัน (หลายคลินิก) — จัดการยังไง?
  • ปิด Visit เมื่อไร / ใครปิด?
ทำไม: Visit คือ key ผูกคิวกับ HIS — flow ทั้งหมดขึ้นกับเรื่องนี้

E · สิทธิ์ สปสช.

E1
เมื่อตรวจสอบสิทธิ์กับ สปสช. แล้ว ต้องส่งอะไรไป HIS เพื่อเปิด Visit?ของคุณ
  • ผลตรวจสิทธิ์คืนอะไรบ้าง (main/sub scheme, รพ.หลัก/รอง, ความถูกต้อง)?
  • ตอนเปิด Visit ต้องแนบ eligibility code รูปแบบไหน?
  • ตรวจสิทธิ์ ผ่าน HIS หรือ ต่อ NHSO ตรง? ใช้ authentication code / เครื่องอ่านบัตรรุ่นไหน/SDK อะไร?
  • กรณีหลายสิทธิ์ — ใครเลือก (ผู้ป่วย/เจ้าหน้าที่)?
ทำไม: เป็นจุดเชื่อม 3 ฝ่าย (คิว-NHSO-HIS) ที่ซับซ้อนสุด ต้องรู้ลำดับ/ฟอร์แมตจริง

F · คิว · เส้นทางบริการ · ความเร่งด่วน

F1
เส้นทางบริการของผู้ป่วยแต่ละแผนกเป็นอย่างไร (ผังบริการ)?เพิ่มเติม
  • "เสร็จจุดนี้ → ไปจุดไหนต่อ" ของแต่ละแผนกเป็นยังไง? (input ของ Service Routing)
  • มีเส้นทางวนกลับ (เช่น Lab → กลับห้องตรวจ) ไหม?
ทำไม: เป็นข้อมูลตั้งค่ากฎ Routing — ถ้าไม่รู้ ตั้งกฎไม่ได้
F2
เกณฑ์ความเร่งด่วน / ER Triage ใช้ระบบอะไร?เพิ่มเติม
  • ใช้ Triage score มาตรฐานไหน (เช่น ESI / MOPH 5 ระดับ)?
  • คิวลัดมีกฎอย่างไร?
ทำไม: ออกแบบ Priority/Triage (TOR 3.3.2/3.3.10)

G · จ่ายยา (Smart Dispensing)

G1
ขอบเขต Smart Dispensing แค่ไหน?ของคุณ
  • แค่ "ส่งคิวไปห้องจ่ายยา" หรือรวม "จัดส่งยาถึงบ้าน"?
  • เชื่อมระบบเภสัช/คลังยาไหม หรือแค่เรียกคิว?
  • จัดส่ง: พาร์ทเนอร์ขนส่งเจ้าไหน? ครอบคลุมพื้นที่ไหน? ใครจ่ายค่าส่ง?
  • Pickup Code ใช้ยืนยันตัวตนยังไง? ใครสแกน?
  • รายงานจัดส่ง/นโยบายโทรแจ้งก่อนส่ง ต้องมีไหม (TOR 3.8.8-9)?
ทำไม: ขอบเขตต่างกันมากระหว่าง "เรียกคิวยา" กับ "logistics ส่งยา" — กระทบ effort สูง

H · Tele-pharmacy

H1
ขอบเขต Tele-pharmacy แค่ไหน?ของคุณ
  • แค่ เติมยา (refill) ออนไลน์ หรือรวม ปรึกษาเภสัชกร (video/chat)?
  • ใครอนุมัติการ refill (เภสัช/แพทย์)? มีขั้นตอนตรวจสอบไหม?
  • รายการยาที่ refill ได้ ดึงจากไหน (HIS/แพทย์อนุญาต)?
  • ใช้ระบบชำระเงิน + จัดส่ง ชุดเดียวกับ Smart Dispensing ไหม?
ทำไม: ถ้ารวม video consult = งานใหญ่ขึ้นมาก ต้องชี้ขอบเขตให้ชัด

I · ระบบชำระเงินออนไลน์

I1
ใช้ผู้ให้บริการ (Payment Gateway) เจ้าไหน?ของคุณ
  • เจ้าไหน (2C2P / GB Prime Pay / ธนาคาร / PromptPay / TrueMoney...)? มีสัญญาอยู่แล้วไหม?
  • เชื่อม ผ่าน HIS billing หรือ ต่อ GW ตรง?
  • ช่องทางที่รองรับ (บัตร/QR/e-wallet)?
  • ใบเสร็จ/ใบกำกับภาษี ออกจากระบบไหน?
  • การกระทบยอด (reconcile) / คืนเงิน (refund) จัดการยังไง?
ทำไม: แต่ละ GW มี API/ขั้นตอนต่างกัน + เรื่องการเงินมี compliance

J · ระบบแจ้งเตือน (LINE OA)

J1
มี event แจ้งเตือนอะไรบ้าง?ของคุณ

เสนอ event ที่ควรมี (ให้ รพ. ยืนยัน/เพิ่ม):

  • คิวใกล้ถึง (เหลืออีก N คิว) · ถูกเรียก · ถูกเรียกข้าม
  • ยืนยันนัดหมาย · เตือนนัดล่วงหน้า (1 วัน/ชม.)
  • ผลตรวจพร้อม · ยาพร้อมรับ/จัดส่ง + Pickup Code
  • ชำระเงินสำเร็จ · ยืนยันลงทะเบียน
  • มี LINE OA channel แล้วหรือยัง? ใครถือ credential?
  • ผูก lineUserId กับผู้ป่วยยังไง (สแกน QR / ลงทะเบียน LIFF)?
  • ความถี่/เวลาส่ง · ขอบเขต Auto-reply · Flex Message template ใครออกแบบ?
ทำไม: กำหนด trigger + ต้องมี channel/consent (PDPA) ก่อนส่งได้

K · นัดหมายผู้ป่วย (เจ้าหน้าที่ + ออนไลน์)

K1
การนัดหมายต้องมีผู้อนุมัติหรือไม่?ของคุณ
  • นัดออนไลน์ของผู้ป่วย ต้องให้เจ้าหน้าที่อนุมัติก่อน หรือ auto-confirm?
  • มีขั้นตอน สอบถามอาการก่อนยืนยัน ไหม (TOR 3.5.1.7)?
  • เชื่อม ระบบนัดเดิมของ รพ. หรือสร้างใหม่?
  • slot/quota ต่อแพทย์/วัน ใครกำหนด? นโยบายยกเลิก/เลื่อนนัด?
ทำไม: มี/ไม่มี approval flow = ออกแบบ workflow + สถานะต่างกัน

L · ฮาร์ดแวร์ / จอแสดงผล / บัตรคิว

L1
จำนวนและรูปแบบอุปกรณ์?เพิ่มเติม
  • จำนวน KIOSK / ตู้ Check-in / จอ TV / เคาน์เตอร์ กี่ชุด?
  • เครื่องอ่านบัตร / printer / กล้อง รุ่น/SDK อะไร?
L2
จอแสดงคิว & บัตรคิว ต้องการแบบไหน?เพิ่มเติม
  • จอแต่ละแผนก layout ต่างกันไหม? เสียงเรียก (ภาษา/รูปแบบ/เรียกชื่อ)?
  • บัตรคิว: ข้อมูลที่ต้องมี, โลโก้, ภาษา, QR? มี ใบนำทาง ไหม?
ทำไม: กระทบ UI จอ/Kiosk + template พิมพ์ (TOR 3.1.6/3.3.6/3.4.1)

M · ปริมาณ / Deploy / Security / PDPA

M1
ปริมาณการใช้งาน & การ deploy?เพิ่มเติม
  • ผู้ป่วย/วัน, peak/ชั่วโมง, จำนวน concurrent users?
  • Deploy on-premise หรือ cloud? สเปก server? ใครดูแล? network ภายใน รพ.?
ทำไม: sizing/performance + วางสถาปัตยกรรม deploy
M2
นโยบาย PDPA / Security?เพิ่มเติม
  • การจัดการ consent ผู้ป่วย (โดยเฉพาะ LINE/ข้อมูลสุขภาพ)?
  • ระดับ audit log ที่ต้องการ? เก็บข้อมูลที่ฝั่งคิวได้แค่ไหน?
ทำไม: ข้อมูลสุขภาพเป็นข้อมูลอ่อนไหว — ต้องออกแบบตั้งแต่ต้น

N · ส่งมอบ / Migration / อบรม / รับประกัน

N1
การส่งมอบและดูแลต่อเนื่อง?เพิ่มเติม
  • ต้อง migrate ข้อมูล จากระบบคิวเดิมไหม?
  • แผน rollout — นำร่องแผนกไหนก่อน?
  • อบรมกี่คน/กี่รอบ? คู่มือ (ไทย/Quick/Technical)?
  • Support SLA ช่วงรับประกัน (เวลาตอบสนอง/Remote)?
ทำไม: ตรง TOR ข้อ 5/6 (รับประกัน/อบรม/120 วัน) + กันงานช่วงส่งมอบ
เคล็ดลับการใช้: ส่งคำถามชุดนี้ให้ รพ./ทีม HIS เป็นลายลักษณ์อักษร และให้ตอบกลับเป็นเอกสาร — คำตอบ (โดยเฉพาะหมวด A, B) จะกลายเป็น ขอบเขตงาน + เงื่อนไข dependency ในสัญญา โดยตรง · เชื่อมโยงกับ Checklist ในเอกสาร HIS Integration Design §8
