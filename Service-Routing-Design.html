🔀 Service Routing Design — เส้นทางบริการอัตโนมัติ

ออกแบบฟีเจอร์ "ตั้งค่าว่าเสร็จจุดนี้ → ไปจุดไหนต่อ" พร้อมเงื่อนไขจาก HIS (เช่น มีสั่งยา → เพิ่มจุดห้องจ่ายยา)

ส่วนขยายของระบบ HAOS Queue · ต่อยอดจากฟีเจอร์ Forward เดิม · 2026-06-26

1. ปัญหา & แนวคิด

ปัจจุบัน: ระบบมีฟีเจอร์ Forward (ส่งต่อ) แต่เป็นการ เลือกปลายทางเองทุกครั้ง (manual) โดยเจ้าหน้าที่ — ทำให้:

  • เจ้าหน้าที่ต้องจำว่า "เสร็จจุดนี้ต้องไปไหนต่อ" เอง → ผิดพลาดได้
  • ไม่รู้อัตโนมัติว่าผู้ป่วยต้องไปห้องจ่ายยาไหม (ขึ้นกับว่าแพทย์สั่งยาหรือเปล่า ซึ่งข้อมูลอยู่ใน HIS)
  • ผู้ป่วยไม่เห็นภาพรวมว่าตัวเองเหลืออีกกี่จุด

สิ่งที่ต้องการ: หน้าตั้งค่า "กฎเส้นทาง (Routing Rules)" ที่กำหนดล่วงหน้าว่าแต่ละจุดเสร็จแล้วไปต่อที่ไหน — และบางเส้นทางเป็น เงื่อนไขจาก HIS เช่น "ถ้า HIS แจ้งว่ามีการสั่งยา → เพิ่มจุดห้องจ่ายยาในเส้นทางอัตโนมัติ"

สรุปแนวคิด: เปลี่ยนจาก "ส่งต่อแบบเลือกเอง" → เป็น "เส้นทางอัตโนมัติตามกฎ + เงื่อนไขจาก HIS" โดยยังเก็บโหมด manual ไว้เป็นทางเลือก

2. ภาพรวม — Care Pathway (เส้นทางการรับบริการ)

มองคิว 1 ใบเป็น "การเดินทางผ่านหลายจุดบริการ" โดยแต่ละจุดเมื่อเสร็จ ระบบจะคำนวณ "จุดถัดไป" จากกฎที่ตั้งไว้ + ข้อมูลจาก HIS แล้วพาคิวเข้าคิวจุดต่อไปให้อัตโนมัติ

3. รูปแบบการ routing 3 แบบ

โหมดคือตัวอย่าง
เสมอ (Always)เสร็จจุดนี้ → ไปจุดถัดไปทุกครั้ง อัตโนมัติซักประวัติ → ห้องตรวจ (เสมอ)
เงื่อนไข HISไปจุดถัดไป เฉพาะเมื่อ HIS แจ้งสัญญาณที่กำหนดห้องตรวจ → ห้องจ่ายยา ถ้ามี DRUG_ORDER
เลือกเอง (Manual)ระบบเสนอตัวเลือก ให้เจ้าหน้าที่เลือกปลายทาง (เหมือน Forward เดิม)กรณีไม่เข้ากฎใด — เจ้าหน้าที่ตัดสินใจ
💡 หนึ่งจุดมีได้หลายกฎพร้อมกัน เช่น ห้องตรวจอาจมีทั้ง "ถ้าสั่ง Lab→Lab", "ถ้าสั่งยา→จ่ายยา", "ถ้าไม่มี→จบ" — ระบบประเมินทุกกฎแล้วสร้างเส้นทางจริง

4. หน้าตั้งค่า (Config UI)

หน้าใหม่ในเมนู Admin/Master: "ตั้งค่าเส้นทางบริการ (Service Routing)" — เป็นตารางกฎ เพิ่ม/ลบ/แก้ได้

⚙️ ตั้งค่าเส้นทางบริการ — ตัวอย่างกฎ
จุดที่เสร็จ (จาก)เงื่อนไขไปต่อที่ (ปลายทาง)ลำดับโหมด
ซักประวัติเสมอห้องตรวจ1อัตโนมัติ
ห้องตรวจถ้ามี LAB_ORDERห้อง Lab1อัตโนมัติ
ห้องตรวจถ้ามี XRAY_ORDERห้อง X-ray2อัตโนมัติ
ห้องตรวจถ้ามี DRUG_ORDERการเงิน → ห้องจ่ายยา3อัตโนมัติ
ห้องตรวจถ้าไม่มี orderจบบริการ9อัตโนมัติ
ห้อง Labเสมอห้องตรวจ (วนกลับ)1อัตโนมัติ

ฟิลด์ในแต่ละกฎ

  • จุดที่เสร็จ (From) — เลือกห้อง + ประเภทบริการ (อ้างอิง master เดิม)
  • เงื่อนไข — เสมอ / ถ้ามีสัญญาณ HIS (เลือกชนิด: DRUG/LAB/XRAY/หัตถการ/custom) / เลือกเอง
  • ปลายทาง (To) — ห้อง + บริการถัดไป (เลือกได้หลายจุดเป็นลำดับ เช่น การเงิน→จ่ายยา)
  • ลำดับ (Priority) — กรณีหลายกฎเข้าเงื่อนไขพร้อมกัน จะจัดคิวตามลำดับนี้
  • โหมด — อัตโนมัติ (auto-forward ทันที) หรือ ต้องให้เจ้าหน้าที่กดยืนยัน
  • เปิด/ปิด — เปิดใช้งานกฎ
💡 แนะนำมีโหมด "แสดงผังเส้นทาง (Preview)" — วาด node ให้เห็นภาพว่ากฎที่ตั้งไว้ทำให้ผู้ป่วยเดินทางยังไง ป้องกันตั้งกฎวนลูปไม่รู้จบ

6. การเชื่อม HIS — HIS บอกว่ามี order อะไร

มี 2 วิธีให้ HIS แจ้งสัญญาณ (ใช้ร่วมกันได้):

วิธี A — HIS Push (Webhook) แนะนำ

เมื่อแพทย์สั่งยา/Lab/X-ray ใน HIS → HIS ยิง webhook มาที่ระบบคิว

// POST /queue/webhooks/order-created  (HIS → Queue)
{
  "eventId": "evt-...",       // idempotency
  "vn": "VN-2026-0099",
  "hn": "00123456",
  "orderType": "DRUG",       // DRUG / LAB / XRAY / PROCEDURE
  "orderedAt": "2026-06-26T10:15:00+07:00"
}

→ ระบบคิวบันทึก TbQueueOrderFlag ของคิวนั้น (DRUG = true)

วิธี B — Queue Pull (ตอนกดเสร็จ)

เมื่อเจ้าหน้าที่กด "เสร็จ" ที่ห้องตรวจ → ระบบคิวเรียก HIS ถาม order ของ Visit นั้น

// GET /his/visits/{vn}/orders  (Queue → HIS)
{
  "vn": "VN-2026-0099",
  "orders": [
    { "type": "DRUG", "status": "active" },
    { "type": "LAB",  "status": "active" }
  ]
}
เลือกแบบไหน? ถ้า HIS รองรับ webhook → ใช้ วิธี A (real-time, ตอนกดเสร็จรู้ทันที) · ถ้าไม่รองรับ → ใช้ วิธี B ถาม ณ ตอนกดเสร็จ (ดูหัวข้อ "ระดับจำเป็น & ทางเลือก" ในเอกสาร HIS Integration)
ทางเลือกสำรอง (ไม่มี HIS): ให้เจ้าหน้าที่ห้องตรวจ ติ๊กเองว่า "สั่งยา / สั่ง Lab" ตอนกดเสร็จ → ระบบ routing ทำงานต่อได้เหมือนกัน (manual flag แทนสัญญาณ HIS)

★ order มาแล้ว ระบบรู้ได้ยังไงว่าต้องไปจุดไหน?

หัวใจสำคัญ: HIS ส่งมาแค่ "ประเภท order" (เช่น DRUG) — HIS ไม่รู้จักและไม่บอกว่า "ห้องจ่ายยา" ของระบบคิวคือห้องไหน เพราะรหัสห้อง/จุดบริการเป็นของระบบคิว/โรงพยาบาลแต่ละที่ การแปลง "DRUG → ห้องจ่ายยา" เกิดจากตารางตั้งค่าฝั่งระบบคิวเอง

มีการแปลง 2 ชั้น ก่อนจะรู้ว่าไปจุดไหน:

ชั้นที่ 1 — รหัส order ของ HIS → ประเภทมาตรฐาน (Category)

HIS อาจส่งรหัสละเอียด (เช่น "MED-ORAL", "LAB-CBC", "XR-CHEST") ต้องจับกลุ่มเป็นประเภทมาตรฐานที่ระบบคิวเข้าใจก่อน

รหัส/ประเภทจาก HISจับเป็น Category
MED-*, DRUG, ยาDRUG_ORDER
LAB-*, ตรวจเลือดLAB_ORDER
XR-*, CT, MRIXRAY_ORDER

* ถ้า HIS ส่ง orderType เป็นประเภทกว้างอยู่แล้ว (DRUG/LAB/XRAY) ก็ข้ามชั้นนี้ได้

ชั้นที่ 2 — Category → จุดบริการปลายทาง (ตารางตั้งค่าที่ admin กำหนด)

นี่คือจุดที่ตอบคำถาม — admin ตั้งค่าไว้ในตารางว่าแต่ละประเภทไปจุดไหน:

⚙️ Order Type → จุดปลายทาง (Order Routing Map)
Category (จาก HIS)จุดปลายทางในระบบคิวหมายเหตุ
DRUG_ORDERการเงิน → ห้องจ่ายยาชำระเงินก่อนรับยา
LAB_ORDERห้อง Lab → วนกลับห้องตรวจเจาะเลือดแล้วกลับพบแพทย์
XRAY_ORDERห้อง X-ray → วนกลับห้องตรวจ
💡 ตารางนี้คือ กฎ Routing (TbServiceRoute) ที่ออกแบบไว้ในหัวข้อ 4–5 นั่นเอง — ฟิลด์ ConditionKey = DRUG_ORDER + ToRoom = ห้องจ่ายยา · พูดอีกแบบ: "ความรู้ว่า DRUG ไปไหน" เก็บอยู่ในกฎที่ admin ตั้ง ไม่ได้มาจาก HIS

กรณีพิเศษที่ต้องตัดสินใจ

กรณีทางออก
มีห้องจ่ายยาหลายห้อง (เช่น แยกชั้น/แยกสิทธิ์)เพิ่มเงื่อนไขในกฎ เช่น DRUG_ORDER + สิทธิ์=ชำระเอง → ห้องจ่ายยา B หรือเลือกตามห้องตรวจต้นทาง
DRUG ปลายทางต่างกันตามแผนกต้นทางใช้ฟิลด์ From ในกฎ (ห้องตรวจ A + DRUG → จ่ายยา A)
HIS ส่งรหัสที่ยัง map ไม่ได้เข้าโหมด เลือกเอง ให้เจ้าหน้าที่ตัดสิน + log ไว้เพื่อเพิ่ม mapping ภายหลัง

8. ตรรกะ Routing Engine (ตอนกด "เสร็จ")

  1. เจ้าหน้าที่กด เสร็จสิ้น ที่จุดปัจจุบัน → ปิด TbQueueService เดิม (Completed)
  2. Engine โหลดกฎทั้งหมดที่ From = จุดปัจจุบัน และ Status = เปิด
  3. ประเมินแต่ละกฎ:
    • ALWAYS → เข้าเงื่อนไขเสมอ
    • HIS_FLAG → เข้าเงื่อนไขถ้าคิวนี้มี TbQueueOrderFlag ตรง ConditionKey
    • MANUAL → แสดงตัวเลือกให้เจ้าหน้าที่เลือก
  4. รวมปลายทางที่เข้าเงื่อนไข เรียงตาม Sequence → สร้าง itinerary (รายการจุดที่ต้องไป)
  5. สร้าง TbQueueService ใหม่ที่จุดแรกของ itinerary (status รอเช็คอิน/รอเรียก) → คิวเข้าคิวจุดถัดไปอัตโนมัติ
  6. ถ้าไม่มีกฎใดเข้าเงื่อนไข → จบ Care Pathway (คิวเสร็จสมบูรณ์)
  7. ⚡ ยิง SignalR อัปเดตจอ/มือถือ ทุกครั้ง (เหมือน Forward เดิม)
ต้องกันลูปไม่รู้จบ: เช่น Lab → ห้องตรวจ → Lab → ... ควรมีกลไกตรวจ (เช่น order ที่ทำแล้วถูก mark completed ไม่ trigger ซ้ำ, หรือจำกัดจำนวนรอบ)
