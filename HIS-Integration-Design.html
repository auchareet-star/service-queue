1. ภาพรวมสถาปัตยกรรม (Integration Architecture)

หลักการ: ระบบคิว ไม่ต่อตรงเข้า Database ของ HIS แต่คุยผ่าน API Gateway เป็นชั้นกลาง (ตามที่ TOR ข้อ 3.1.3 / 3.2.4 / 3.6.3 ระบุ "API GATEWAY หรือตามที่โรงพยาบาลกำหนด") เพื่อความปลอดภัย, ควบคุมสิทธิ์, และลดการผูกติดกับ schema ภายใน HIS

2 ช่องทางการสื่อสาร: (1) Sync REST — ระบบคิวเป็นฝ่ายเรียก HIS เมื่อต้องการข้อมูลทันที (ค้นผู้ป่วย, ตรวจสิทธิ์, เปิด Visit)  |  (2) Async Event/Webhook — สำหรับการแจ้งเหตุการณ์สองทาง (HIS แจ้งว่ามีนัดใหม่ → Queue, Queue แจ้งว่าเรียกคิว/เสร็จ → HIS/LINE)

2. รูปแบบการเชื่อมต่อ 3 แบบ (เลือกตามลักษณะข้อมูล)

รูปแบบใช้เมื่อตัวอย่างใน TORข้อดี/ข้อควรระวัง
① Synchronous REST
(Request/Response)		ต้องการข้อมูลทันทีตอนผู้ป่วยยืนอยู่หน้าตู้/เคาน์เตอร์ค้นผู้ป่วยจากบัตร ปชช., ตรวจสิทธิ์ สปสช., เปิด Visit, ดึงรายการนัดวันนี้ง่าย ตรงไปตรงมา · ต้องกำหนด timeout + fallback ถ้า HIS ช้า/ล่ม
② Async Event / Webhook
(Publish/Subscribe)		เหตุการณ์เกิดฝั่งหนึ่ง ต้องแจ้งอีกฝั่งโดยไม่ต้องรอHIS จองนัด → activate คิวอัตโนมัติ (3.5.1.5), คิวถูกเรียก/เสร็จ → แจ้ง HIS + LINE OAทนทาน, decoupled · ต้องมี retry + idempotency + dead-letter
③ Polling (สำรอง)HIS ไม่รองรับ push และต้องติดตามสถานะที่เปลี่ยนช้าสถานะผล LAB/X-ray (3.3.7) ถ้าไม่มี eventใช้เมื่อจำเป็นเท่านั้น · เปลือง resource — กำหนดความถี่ให้เหมาะ
คำแนะนำ: รูปแบบที่ดีที่สุดขึ้นกับว่า HIS รองรับอะไร — โรงพยาบาลไทยส่วนใหญ่ (เช่น HOSxP, JHCIS, SSB, Vendor เฉพาะ) มักเปิด REST/JSON หรือ HL7 v2 / FHIR ผ่าน middleware ควรยืนยันกับทีม HIS ว่ารองรับ event/webhook หรือไม่ ถ้าไม่ ให้ออกแบบ polling สำรอง

3. API ที่ HIS ต้องเปิดให้ (HIS → ระบบคิวเรียกใช้)

นี่คือรายการ endpoint ที่ HIS ต้องเปิดให้ระบบคิวเรียก (ผ่าน Gateway) จัดกลุ่มตามภารกิจ พร้อมแมปกับข้อ TOR

★ ระดับความจำเป็นของแต่ละ API + ทางเลือกเมื่อ HIS ไม่เปิดให้

ไม่ใช่ทุก API ที่ "ต้องมี" — บางอย่างถ้า HIS ไม่เปิด ก็ยังมี ช่องทางอื่นทดแทนได้ ตารางนี้แยกระดับความจำเป็นและบอกทางออกแต่ละข้อ

จำเป็น ขาดไม่ได้ถ้าจะเชื่อม HIS จริง (เลี่ยงได้แค่ทำ manual/standalone) ควรมี ตรง TOR; ถ้าไม่มีพอมีทางเลือกที่ยอมรับได้ ทางเลือก ใช้ช่องทางอื่นแทนได้ง่าย ไม่ต้องให้ HIS เปิด API

กลุ่ม ⑤ Real-time (Webhook vs Polling)

ความสามารถระดับทางเลือก/ช่องทางแก้
HIS push event (webhook) เช่น appointment-created, order-statusทางเลือกถ้า HIS ไม่รองรับ push: ใช้ Polling เรียก API เช็คเป็นรอบ ๆ แทน — ได้ผลเหมือนกันแต่หน่วงและกิน resource มากกว่า กำหนดความถี่ให้พอเหมาะ
สรุปชุด API ขั้นต่ำสำหรับเฟส 1 (MVP เชื่อม HIS): ต้องมีจริง ๆ แค่ ① ค้นผู้ป่วย/ได้ HN และ ② ผูก Visit (VN) — สองตัวนี้ทำให้คิวเชื่อมกับ HIS ได้ ส่วนข้อมูลคลินิก/เภสัช/การเงิน ใช้ deep-link ไป HIS + manual + polling ไปก่อน แล้วค่อยอัปเกรดเป็น API จริงในเฟสถัดไป
หลักคิด: ทุกครั้งที่ HIS ไม่เปิด API ให้ มี 4 ทางออกมาตรฐาน → (1) อ่านบัตร/ข้อมูลในเครื่อง · (2) Deep-link/SSO ไปหน้า HIS · (3) Manual โดยเจ้าหน้าที่ · (4) Batch import / Polling เลือกตามว่าต้องการความ real-time แค่ไหน

4. API / Event ที่ระบบคิวต้องเปิดให้ (Queue → HIS / ระบบอื่นเรียกใช้)

เพื่อให้เป็นการเชื่อม สองทาง ระบบคิวต้องเปิดบริการกลับด้วย 2 กลุ่ม:

4.1 Webhook Receiver (HIS/ระบบอื่น ส่งเข้ามาที่คิว)

MethodEndpoint (ที่คิวเปิด)หน้าที่TOR
POST/queue/webhooks/appointment-createdHIS แจ้งว่ามีการจองนัด → ระบบคิว activate คิวอัตโนมัติ3.5.1.5
POST/queue/webhooks/order-createdHIS แจ้งว่าแพทย์สั่ง ยา/Lab/X-ray (orderType) → ตั้ง flag เพื่อให้ Routing Engine เพิ่มจุดถัดไป (เช่น เพิ่มห้องจ่ายยา)3.3.9, routing
POST/queue/webhooks/order-statusHIS แจ้งสถานะผล LAB/X-ray เปลี่ยน → ปลดล็อค/จัดลำดับคิว3.3.7
POST/queue/webhooks/payment-completedPayment/HIS แจ้งชำระเงินแล้ว → ส่งคิวไปจุดจัดยาอัตโนมัติ3.8.5, 3.9.6
🔀 order-created เป็นหัวใจของฟีเจอร์ เส้นทางบริการอัตโนมัติ (Service Routing) — HIS แจ้งว่ามีสั่งยา/Lab/X-ray แล้วระบบคิวเพิ่มจุดที่ต้องไปต่อให้เอง · รายละเอียดเต็มดูที่เอกสาร Service Routing Design (หัวข้อ 6)

6. Master Data Mapping (จุดที่มักพลาด)

ก่อนเชื่อมได้ ต้อง จับคู่รหัส master ระหว่างระบบคิวกับ HIS ให้ตรงกัน มิฉะนั้นคิวจะวิ่งผิดแผนก

ระบบคิว (HAOS)HISวิธีจับคู่
TbmRoom (ห้อง/จุดบริการ) + CodeClinic / Department / Ward codeเพิ่มฟิลด์ HisClinicCode ใน Room เพื่อ map 1:1
TbmServiceType (ประเภทบริการ)Service / Clinic sub-typeเพิ่ม HisServiceCode
Queue / TicketVisit (VN) + HNเก็บ HN, VN, AppointmentId ใน TbQueue/TbQueueService
ServiceGroupสิทธิ์ (Eligibility/Payer)เก็บ EligibilityCode ที่ผู้ป่วยเลือก
Priority (ใหม่)Triage level / urgencyเพิ่มฟิลด์ PriorityLevel รองรับคิวลัด/ER
คิดเพิ่ม: ควรทำตาราง mapping เป็น config (UI ตั้งค่าได้) ไม่ hardcode เพราะรหัสคลินิก/แผนกแต่ละโรงพยาบาลไม่เหมือนกัน และอาจเปลี่ยน — สอดคล้องกับ TOR ที่ย้ำ "กำหนดโดยผู้ควบคุมระบบของโรงพยาบาล"

7. Security & PDPA (ข้อมูลสุขภาพ= ข้อมูลอ่อนไหว)

  • API Gateway เป็นด่านเดียว: OAuth2 Client Credentials หรือ API Key ต่อระบบ + mTLS ระหว่างคิว ↔ HIS
  • IP Allowlist + Rate limiting + WAF ที่ Gateway
  • Audit log ทุก request ที่แตะข้อมูลผู้ป่วย (ใคร/เมื่อ/ดึงอะไร) — ระบบมี RequestResponseLoggingMiddleware อยู่แล้ว ต่อยอดได้
  • PDPA / ข้อมูลสุขภาพ: ดึงข้อมูลเท่าที่จำเป็น (data minimization), ไม่เก็บข้อมูลทางการแพทย์ถาวรในฝั่งคิวถ้าไม่จำเป็น (ดึง on-demand แล้วทิ้ง), เข้ารหัสข้อมูลพักชั่วคราว, มีการจัดการความยินยอม (consent) สำหรับ LINE OA
  • Idempotency ทุก webhook (eventId) กัน event ซ้ำ + retry with backoff + dead-letter queue
  • Token จาก KIOSK/ตู้ ควรเป็น device token แยก ไม่ใช้สิทธิ์ผู้ใช้ทั่วไป

คำถามเหล่านี้ชี้ขาด scope/ความเสี่ยง ควรถามให้ครบก่อนประเมินงาน:

#คำถามทำไมสำคัญ
1HIS ยี่ห้ออะไร? (HOSxP / JHCIS / SSB / Vendor เฉพาะ) และมี API Gateway/middleware อยู่แล้วไหมกำหนดมาตรฐาน (REST/HL7/FHIR) และความพร้อม

4รองรับ Webhook/Event push ไหม หรือทำได้แค่ให้ polling?กำหนดสถาปัตยกรรม async vs polling
